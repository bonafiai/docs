One app, one deploy, hard module boundaries. Every feature lands in the same shape, so anyone (human or agent) knows where things live without asking.

## Modules

A module is a domain folder in `lib/` that owns its logic, its Hono router, its jobs, and its schema slice. Modules talk through exported functions and events, never through another module's tables.

```text
lib/
  sources/           one domain module
    router.ts        its Hono routes, mounted by the catch-all
    jobs/            its Inngest functions (thin shells)
    schema.ts        its Drizzle tables
    parse.ts         its domain logic (fat modules)
  review/
  entities/
  connectors/
  shared/            events registry, tenant db, gateway client
```

## App

The Next.js layout we always follow:

```text
app/
  (app)/             route group, the authenticated product
    review/
      page.tsx
      _components/   colocated UI, not routable
      _lib/          colocated helpers, not routable
  api/
    [[...route]]/    one catch-all, mounts every module router
  layout.tsx
proxy.ts             the edge session gate
```

| Name | Description |
| --- | --- |
| Route groups | `(app)` and friends group routes without affecting URLs |
| Parallel routes | Used where one view renders independent panes, never for layout convenience |
| `_components`, `_lib` | Private folders, colocated with the route that uses them, excluded from routing |
| Catch-all API | One `app/api/[[...route]]`, Hono owns everything under `/api` |

## Boundaries

- UI imports from `lib/<module>`, never the other way around.
- A module never queries another module's tables, it calls the exported function or listens for the event.
- `shared/` is small on purpose: the event registry, the tenant transaction, the gateway client. If it grows, something is misplaced.
