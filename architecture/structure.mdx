One app, one deploy, hard module boundaries. Every feature lands in the same shape, so anyone (human or agent) knows where things live without asking. This is the tree, the sections below say why each zone exists.

```text
app/                  routes only, no business logic
  (app)/              route group, the authenticated product
    review/
      page.tsx
      _components/    UI private to this route
      _lib/           helpers private to this route
  api/
    [[...route]]/     one catch-all, Hono owns /api
  layout.tsx
lib/                  all business logic, one folder per domain
  sources/
    router.ts         its Hono routes
    schema.ts         its Drizzle tables
    types.ts          hand-written domain types only
    jobs/             its Inngest functions, crons included
    parse.ts          its domain logic
  shared/             events registry, tenant db, gateway client
proxy.ts              the edge session gate
specs/                feature specs from the loop
.specify/             spec-kit memory, the constitution
```

## Modules

A domain folder in `lib/` owns its logic, router, jobs, tables, and types. Modules talk through exported functions and events, never another module's tables. New domain, new folder, same shape, per [Hono best practices](https://hono.dev/docs/guides/best-practices): modular sub-routers over one giant app.

## Types

Derive, don't hand-write. Table types come from Drizzle (`$inferSelect`, `$inferInsert`) and live in `schema.ts`, see [schema declaration](https://orm.drizzle.team/docs/sql-schema-declaration). Event types come from the zod registry via [inference](https://zod.dev/?id=type-inference). API types flow to the client through [Hono RPC](https://hono.dev/docs/guides/rpc), no hand-written DTOs. `types.ts` holds only domain types that cannot be derived. A type needed by two modules moves to `shared/`, and that is a smell to watch.

## Jobs

Every [Inngest function](https://www.inngest.com/docs/features/inngest-functions) lives in the owning module's `jobs/`, one file per job, thin shell delegating to domain logic. A cron is just a job with a [cron trigger](https://www.inngest.com/docs/guides/scheduled-functions), same folder, named for its sweep. All jobs register on the single serve handler in the catch-all.

## App

`app/` is routing, nothing else. [Route groups](https://nextjs.org/docs/app/getting-started/project-structure) organize without touching URLs, `_components` and `_lib` are private folders that colocate route-private code out of the router, and one catch-all hands `/api` to Hono, see [Hono on Vercel](https://hono.dev/docs/getting-started/vercel).

## Shared

Three residents: the event registry, the tenant transaction, the gateway client. Small on purpose, growth means something is misplaced.

## Placement

When in doubt, this is where it goes:

| Name | Description |
| --- | --- |
| A route | `lib/<module>/router.ts`, mounted by the catch-all |
| A table | `lib/<module>/schema.ts`, types derived, not written |
| A job or cron | `lib/<module>/jobs/`, one file, thin shell |
| A domain type | `lib/<module>/types.ts`, only if not derivable |
| Route-private UI | `app/.../_components/`, colocated |
| Anything two modules need | `shared/`, and question it |
