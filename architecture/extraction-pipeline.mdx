title: "Extraction"
description: "Source to truth in seven stages."

A document moves through seven stages from upload to truth. Each stage reads only the previous stage's persisted output, emits exactly one event, and never reaches around its neighbor. Anchoring is not a stage: a document role is a document-scoped resolvement folded into Stage 5.

This page is the canonical design. It records the "how" — storage contracts, event wiring, and the normalizer and composition algorithms. Specs in `specs/NNN-*` are implementation slices that reference this page; the Drizzle tables in `lib/<module>/schema.ts` are the implementation source of truth, and types derive from them.

## Stage map

| # | Stage | Input | Persisted output | Emits |
| --- | --- | --- | --- | --- |
| 1 | Source upload | raw file | R2 object, `sources` row | `source.uploaded` |
| 2 | Parse and classify | source file | `parses` row, artifact in R2, `pages` rows, `doc_type` on the run | `source.parsed` |
| 3 | Structural | parse artifact | `extraction_runs` row, `tables`, `table_axes`, `table_cells`, `quantity_atoms`, `date_atoms`, `labels` | `run.structured` |
| 4 | Non-table | parse artifact, Stage 3 rows | `freestanding_values` (same payload objects) | `run.values_extracted` |
| 5 | Resolvement | labels, precedents, registry | `label_resolvements`, `resolvement_candidates`, `precedents`, document roles | `run.resolved` |
| 6 | Proposals | resolvements, registry | `proposals`, `proposal_inputs` | `run.proposed` |
| 7 | Commit | approved proposals | truth rows, `proposal_decisions` | `proposal.committed` |

**Seven stages, not eight.** Anchoring is not a stage. A document-level role ("this value is the document's investor") is a document-scoped resolvement — the same row that resolves a value label carries an optional `document_role`. Merging it into Stage 5 removes a stage, a table, an event, and a whole selection machine.

## Principles

| Name | Description |
| --- | --- |
| Relational first | Everything derived from a parse is relational rows that reference each other. The raw parse artifact is the one opaque blob, stored in R2. The only JSONB in Postgres is the disposable proposal body, rebuildable from relational inputs. |
| Immutable inputs | Raw files, parse artifacts, and resolvement records are never mutated. Re-running a stage produces new rows. |
| Provenance end to end | Every truth value walks back: truth row to proposal to atom or label to cell to table to parse artifact to source file, with bboxes at every geometric step. |
| One resolvement machine | All labels, metric headers, entity mentions, KV keys, go through one precedent, candidate, reasoning system. One store, one audit shape. |
| Few object classes | Table cells and freestanding values reuse the same payload objects. No parallel near-duplicate concepts. |
| Modular stages | Each stage reads only the previous stage's persisted output and emits one completion event. A stage's implementation can be swapped without touching neighbors. |

<Callout type="warning">
A run is all-or-nothing per stage. A stage advances `status` only on full completion. A mid-stage failure leaves the run `failed` and its partial rows are never consumed. Recovery is a new run, never a resume.
</Callout>

## Event bus

Each stage is an Inngest job triggered by exactly one event and emitting exactly one event. Re-running a stage re-emits its event and the downstream chain replays — no stage ever reaches around its neighbor. A precedent minted in review re-resolves held labels by re-emitting `run.resolved`.

```mermaid
flowchart LR
  A[source.uploaded] --> P["parse + classify"]
  P --> B[source.parsed]
  B --> S[structure]
  S --> C[run.structured]
  C --> V[extract values]
  V --> D[run.values_extracted]
  D --> R["resolve labels + roles"]
  R --> F[run.resolved]
  F --> O[compute proposals]
  O --> G[run.proposed]
  G -.review.-> H[decisions]
  H --> I[proposal.committed]
  X[precedent minted] -.re-resolve held labels.-> F
```

## Run spine

Stages 3 to 6 hang off a thin `extraction_runs` row, not `parse_id` directly. The run records which code version derived which rows from which parse. Fixing a deterministic bug means a new run over the same parse, never re-parsing and never mutating the old run's rows.

**Storage:** `extraction_runs`

```text
extraction_runs
  id                uuid PK
  parse_id          FK parses
  organization_id   FK
  pipeline_version  text        -- code version of the deterministic stages
  doc_type          text (nullable)         -- run-scoped classifier verdict; sole consumer is the Stage 5 scope chain
  classifier_version text (nullable)
  status            enum: running | structured | values_extracted | resolved | proposed | failed
  created_at        timestamptz
```

| Name | Description |
| --- | --- |
| Active pointer | `parses.active_run_id` designates the live run. Superseded runs stay for audit and are never read by downstream consumers. |
| Ownership | Every Stage 3 and later row foreign keys `run_id`. Deleting a superseded run cascades its derivation. |
| All-or-nothing | Downstream only reads runs that reached the required status, following `active_run_id`. A failed run's partial rows are orphaned and swept. |
| Deterministic IDs | Stage 3 uses UUIDv5 over `run_id` plus structural path so re-runs are diffable. LLM stages start fresh. |

## Stages

### 1 Source upload

A user or connector uploads a raw file. The file goes to R2 unmodified. A `sources` row is the provenance root.

- The R2 object is written once. `r2_key` includes the content hash so the same bytes always land on the same key.
- Duplicate upload, same `content_hash` within the org, creates a new `sources` row pointing at the existing R2 object. The pipeline is not re-run; the new source links to the existing parse.
- Duplicate provenance walks to the first uploader by design. The shared parse's `source_id` points at the original source, so walking a truth row back lands on the first uploader. Consumers that need "who most recently uploaded these bytes" query `sources` by `content_hash`, not the derivation walk.

**Storage:** `sources`

```text
sources
  id                uuid PK
  organization_id   FK
  uploaded_by       FK member (nullable, connectors)
  file_name         text
  mime_type         text
  byte_size         int
  content_hash      text        -- sha256 of the raw bytes; duplicate detection
  r2_key            text        -- immutable; never rewritten
  status            enum: uploaded | parsing | parsed | failed
  created_at        timestamptz
```

Emits `source.uploaded`, triggers Stage 2.

### 2 Parse and classify

The parser turns the raw file into a machine-readable artifact stored verbatim in R2. The artifact is the one big opaque blob in the system: a frozen, versioned snapshot of what the parser saw, from which every relational row downstream is deterministically derivable. The `parses` row keeps `artifact_r2_key` and size metadata. The artifact is read through one accessor so the storage backend is swappable.

Classification is run-scoped, not parse-scoped. `doc_type` lives on `extraction_runs` because its only consumer is the Stage 5 scope chain. Reclassifying is a new run over the same artifact, never a mutation of the frozen parse. `doc_type` is a scoping hint, never a hard gate: the scope chain falls through to org and global, so a wrong or missing `doc_type` degrades to broader precedents rather than blocking.

- The artifact is stored verbatim, no cleanup or enrichment. A derived shape is Stage 3's relational output, not a second blob.
- Artifact location: R2 from day one. Four parallel views of an agentic parse of a 50-page document are multiple MB; that never belongs in a Postgres row.
- A source can have multiple parses (re-parse after a parser upgrade). Parses are append-only; `sources.active_parse_id` designates the live one explicitly.
- Pages are first-class. Parse completion writes one `pages` row per page with index and dimensions. Every geometric row downstream foreign keys `page_id`. Bboxes are normalized `[x0, y0, x1, y1]` in `0.0` to `1.0`; the `pages` dimensions map back to PDF points.

**Storage:** `parses`, `pages`

```text
parses
  id                uuid PK
  source_id         FK sources
  parser            text
  parser_version    text        -- version + config hash; identifies determinism envelope
  status            enum: pending | succeeded | failed
  error             text (nullable)
  page_count        int
  artifact_r2_key   text        -- R2 object holding the raw parse output, verbatim
  artifact_bytes    int
  active_run_id     FK extraction_runs (nullable)
  created_at        timestamptz
  completed_at      timestamptz

sources (addition)
  active_parse_id   FK parses (nullable)   -- explicit pointer to the live parse

pages
  id                uuid PK
  parse_id          FK parses
  page_index        int          -- 0-based
  width             numeric      -- PDF points; the bbox coordinate space
  height            numeric
  UNIQUE (parse_id, page_index)
```

#### The parse artifact, by example

The shape below is the target artifact the Stage 2 parser seam produces. It bundles the parser result's parallel views — structured items, grounded per-cell bboxes, markdown, plain text — into one frozen blob. All four views stay in the one artifact because they are projections of the same parse and are only consistent within it. Stage 3 joins views 1 and 2 to build relational geometry; Stage 4 reads view 3. Nothing downstream ever re-calls the parser.

```jsonc
{

  "tier": "agentic_plus",                  // parse tier used (cost/quality knob)
  "version": "0.6.2",
  "pageCount": 4,

  // VIEW 1: layout items per page. Feeds Stage 3 (tables, axes, cells) + Stage 4 (freestanding values).
  "pages": [
    {
      "page_number": 1,
      "page_width": 612, "page_height": 792,   // PDF points; needed to normalize bboxes to [0,1]
      "items": [
        { "type": "heading", "level": 1, "value": "Capital Account Statement",
          "bbox": [{ "x": 43, "y": 51, "w": 320, "h": 22 }] },
        { "type": "text", "value": "Fund: ABC Growth Fund III, L.P.\nInvestor: Jane Doe Family Trust",
          "bbox": [{ "x": 43, "y": 90, "w": 400, "h": 36 }] },     // -> Stage 4 freestanding values
        { "type": "table",

            ["Beginning Capital Balance", "$12,500,000", "$11,800,000"],

    {

  "textByPage":     [ { "page_number": 1, "text": "Capital Account Statement Fund: ABC..." } ],
  "metadata": [ { "page": 1, "confidence": 0.99 } ]
}
```

Emits `source.parsed`, triggers Stage 3.

### 3 Structural

A pure deterministic function of the parse artifact. No LLM, no DB lookups, no network. It joins the artifact's layout view with the grounded sidecar and writes the relational substrate: tables, axes, cells, atoms, labels. Re-running with the same artifact and same `pipeline_version` produces identical rows.

#### Fidelity contract: transcription, not interpretation

This is what keeps a deterministic Stage 3 from breeding a code branch per document layout:

> Stage 3 mirrors exactly what the parser asserts about structure — and asserts nothing of its own.

It transcribes cell grid, spanning-header structure (from the `html` view), indent/level, and grounded bboxes. It never interprets: it does not decide a row is a total, that a table is transposed, that two tables are continuations, or that a blank column is a stub. Those are meanings, resolved at Stage 5 and later. Pathological layouts are absorbed downstream as data, never as Stage 3 code:

| Layout hazard | What Stage 3 does | Where it resolves |
| --- | --- | --- |
| Totals / subtotal row | writes it as an ordinary row + label | its label resolves to a total-kind canonical (Stage 5); proposal sign/aggregation follows the registry |
| Transposed table (metrics on the row axis) | writes symmetric `table_axes` (row/column are one class) | the metric axis is whichever axis's labels resolve to metrics — no orientation flag |
| Continuation table across pages | two separate `tables` rows | identical `(canonical, slots, period)` merge at proposal dedup (Stage 6) |
| Blank stub / unlabeled axis | axis row with `label_id = null` | nothing to resolve; carries no meaning |
| Multi-row spanning header | `parent_axis_id` + `span_end` from the `html` structure | Stage 6 composition walks the axis hierarchy |
| Doubled value in an emphasized cell (`70,727 70,727`) | lexical normalizer collapses byte-identical repeats; differing numbers mint no atom | coverage residue surfaces any dropped cell loudly (Stage 6) |

#### The object model

```text
table --< table_axes (axis: row | column; hierarchical) --> label   (descriptor position)
   |
   +--< table_cells --> row axis, column axis (leaf)
                   +--> exactly one of:  quantity_atom | date_atom | label (value position)
```

- **`table_axes`** — one object class for both rows and column headers. A column header and a row are the same thing structurally: an indexed line of the grid that may carry a label and may sit in a hierarchy. Instead of `table_rows` + `table_columns` + `table_headers`, one class covers all three.
- **Atoms** (`quantity_atoms`, `date_atoms`) — deterministically parsed values. Pure payload: no geometry, no label text, never resolved. Geometry and context live on whatever carries the atom.
- **`labels`** — pure text identity for any string whose meaning requires resolvement. Cross-document identity is the normalized text itself. No geometry — the carrier owns the bbox.

#### The atom/label seam

The discriminator between atom and label is not number-vs-text — it is:

> Does deterministic lexical normalization fully capture the meaning (atom), or does the string need semantic resolvement (label)?

Quantities (`$12,500,000` -> `12500000 USD`) and dates (`3/15/24` -> `2024-03-15`) are atoms. Every other string in a value position is a label, resolved through the same machinery as header labels. A label plays one of two roles, separated structurally by which slot references it — never by a flag on the label:

| | Descriptor label | Value label |
| --- | --- | --- |
| Referenced from | `table_axes.label_id`, `kv.key_label_id` | `table_cells.value_label_id`, `kv.value_label_id`, narrative mention |
| What it is | describes values stored elsewhere | is the value — the surface string is the payload |
| Resolvement supplies | the meaning of other rows' payloads (canonical metric, period, entity axis, measure) | the reference/type of itself (entity link, literal, enum) |
| Default expectation | canonical metric / period / entity / measure | entity mapping; literal/enum are exceptions |

**Storage:** `tables`, `table_axes`, `table_cells`, `quantity_atoms`, `date_atoms`, `labels`

```text
tables
  id                  uuid PK
  run_id              FK extraction_runs
  page_id             FK pages
  table_index         int
  bbox                numeric[4]
  n_rows              int
  n_cols              int
  title_label_id      FK labels (nullable)   -- heading directly above the table; joins the
                                       -- governing chain at composition (allocation tables carry
                                       -- their concept here while rows name only entities)

table_axes
  id                  uuid PK
  table_id            FK tables
  axis                enum: row | column
  axis_index          int
  span_end            int (nullable)         -- spanning header coverage
  level               int                    -- hierarchy depth
  parent_axis_id      FK table_axes (nullable)
  bbox                numeric[4] (nullable)
  label_id            FK labels (nullable)   -- null: an unlabeled axis
  UNIQUE (table_id, axis, level, axis_index)

table_cells
  id                  uuid PK
  table_id            FK tables
  row_axis_id         FK table_axes
  column_axis_id      FK table_axes          -- always a leaf column
  row_index           int
  col_index           int
  raw_text            text
  bbox                numeric[4]
  quantity_atom_id    FK quantity_atoms (nullable)   -- exactly one of the three payload refs set
  date_atom_id        FK date_atoms (nullable)
  value_label_id      FK labels (nullable)
  footnote_marker     text (nullable)
  UNIQUE (table_id, row_index, col_index)

quantity_atoms                         -- a measurable value; never resolved
  id                  uuid PK
  run_id              FK extraction_runs
  raw_text            text
  value               numeric
  unit                text (nullable)        -- USD | EUR | ratio | shares
  normalization_version text

date_atoms                             -- a calendar value; never resolved
  id                  uuid PK
  run_id              FK extraction_runs
  raw_text            text
  value               date
  normalization_version text

labels
  id                  uuid PK
  run_id              FK extraction_runs
  kind                enum: column_header | row_header | cell_text | kv_key | kv_value | narrative | table_title
                      -- structural provenance ONLY, never a role; role derives from FK topology
  surface_text        text
  normalized_text     text                   -- cross-document identity; the precedent match key
  unit_hint           text (nullable)        -- currency/percent affix stripped: "INVESTOR (EUR)" -> unit_hint EUR
  scale_hint          numeric (nullable)     -- "in $1,000s" -> 1000
```

Normalizer rules (all pure, versioned by `normalization_version`, no LLM):

- **Lexical only.** `$5MM -> 5_000_000 USD`, `(93,750) -> -93750`, `3/15/24 -> 2024-03-15`. Interpretation is Stage 5+.
- **Label affixes are extracted, not resolved.** Unit/scale qualifiers strip into `unit_hint`/`scale_hint`; matching is on the remainder, so generic headers become more matchable.
- **Identical repetition collapses.** Byte-identical repeated numeric tokens (`70,727 70,727`) and character-interleaved doubling (`IINNVVEESSTTEEEERRDDEERR`) collapse deterministically. Two differing numbers in one cell never guess: no atom is minted, `raw_text` is kept, and the coverage invariant surfaces it loudly.
- **Number locale is inferred per document.** A document-level pre-pass fixes locale from unambiguous tokens (`X.XXX,XX` vs `X,XXX.XX`); ambiguous tokens then follow it. Intra-number whitespace is cleaned in the same pass.
- **A lone currency token joins the adjacent number.** A cell/token holding only a currency symbol (`$ | 9,907,245.92`) is absorbed as the adjacent atom's unit; a currency-only column mints no atoms.

Emits `run.structured`, triggers Stage 4.

### 4 Non-table

Values outside tables: title-block key-value pairs, amounts and dates in prose, standalone mentions. This stage reuses the Stage 3 payload objects. The only new object is the carrier, `freestanding_values` — a KV pair and a narrative value are the same thing structurally: a value occurrence outside a grid, with an optional key.

- **One LLM pass** over per-page markdown with tables stripped. One prompt version to audit (`extractor_version`).
- **Verbatim guard.** Every extracted span must appear verbatim in the parse artifact. `raw_text` is the proof; `bbox` is recovered by grounding the span against the artifact's grounded token bboxes (not the lossy markdown). An ungroundable span is rejected.
- **Pairing is geometric, not reading-order.** Key-to-value linking uses grounded bboxes (spatial adjacency), never markdown proximity. `pairing_method` records the signal, `pairing_confidence` its strength. A weak or ambiguous pairing becomes a review item, never a silent commit — this is the one place a parser quirk could otherwise produce wrong truth rather than residue.
- **The LLM extracts, it does not interpret.** A key becomes a label like any other and waits for Stage 5. No dedup here; corroboration happens at Stage 6.

**Storage:** `freestanding_values`

```text
freestanding_values
  id                  uuid PK
  run_id              FK extraction_runs
  page_id             FK pages
  key_label_id        FK labels (nullable)   -- null: keyless mention
  key_bbox            numeric[4] (nullable)
  raw_text            text
  bbox                numeric[4]
  quantity_atom_id    FK quantity_atoms (nullable)   -- exactly one payload ref set
  date_atom_id        FK date_atoms (nullable)
  value_label_id      FK labels (nullable)
  pairing_method      enum: geometric | text_order | llm
  pairing_confidence  numeric (nullable)     -- low: a review item, never a silent commit
  extractor_version   text
  context_snippet     text (nullable)
```

Emits `run.values_extracted`, triggers Stage 5.

### 5 Resolvement

Every label — descriptor or value, table or freestanding — gets exactly one resolvement per run through one machine. This stage also assigns document roles: a role is a document-scoped resolvement, not a separate stage or table. The same row that resolves `"Jane Doe Family Trust"` to an entity can carry `document_role = investor`.

#### The universal target

A resolvement (and a precedent) always asserts one thing: a target. The target's kind is what makes the system universal:

| target_kind | target payload | typical label |
| --- | --- | --- |
| `canonical_metric` | base concept, optionally scope-qualified (`capital_called@inception_to_date`) | metric header, KV key like "Capital Called" |
| `entity` | entity id (or a mint request) | fund/investor name |
| `document_ref` | document role name (`investor`, `fund`) | role-selector header (`INVESTOR`, `FUND`) — points at the document's role, names no concrete entity |
| `period` | period strategy (`QUARTER@period_end`, literal date) | date-axis header, "Period Ended" key |
| `measure` | representation: `value` \| `percentage` \| `shares` \| `ratio` | generic value header (`Amount`, `%`) — supplies how a number is represented, never what it is |
| `literal` | the surface string is the value | IBAN, account number, status code |

Three of these compose rather than stand alone:

- **`canonical_metric` targets may carry a temporal scope.** Temporal scope changes what the number is (cumulative capital called != this quarter's call). The registry defines each base concept's allowed scopes (`period`, `year_to_date`, `inception_to_date`, `instant`, `transaction`) as data, not schema. Scopes are context-free, so "before/after this notice" is never a scope — both are `inception_to_date` at two different as-of dates.
- **`document_ref` targets defer to the document's roles.** `INVESTOR (EUR)` names no entity; it dereferences whatever the document's `investor` role resolves to, the symmetric counterpart of `document_role`.
- **`measure` targets carry no meaning of their own.** They attach to whichever base concept governs the same cell at composition. `"amount" -> value`, `"%" -> percentage` are prime global seed precedents.

#### The resolve ladder (per label)

```text
1. PRECEDENT — exact match on normalized_text within the scope chain
   (doc_type -> org -> global); deterministic, auto-accepted
2. CANDIDATE GENERATION — mechanical, non-LLM: registry similarity (metrics),
   blocking + similarity vs entities + aliases, period/measure recognizers
3. REASONING — LLM picks among the enumerated candidates only (or abstains);
   result is PROPOSED, prefilled in review, never auto-accepted
4. HUMAN — confirms / overrides in review; confirmation writes a precedent (the flywheel)
```

There is deliberately no positional/structural match tier. A label's cross-document identity is its normalized text plus the document's `doc_type` — the same key whether it came from a table header, a KV key, or prose. Positional context is used compositionally in Stage 6, keeping precedents robust to layout drift.

An exact match (step 1, the only auto-accepting tier) fires only when all three hold: exactly one non-revoked precedent matches the `normalized_text` at the winning scope tier (string equality, never similarity); its scope applies to this document (`doc_type -> org -> global`, most specific first); and the target is still valid. Conflicts fall through to candidates, never guess. Bare/generic labels ("Amount", "Total", "%") must never bind to a base concept — their correct resolvement is a `measure` target.

#### Document roles (the former anchoring stage)

A document role elects one already-resolved occurrence as the document's default for that role. It rides on the same resolvement rows via a `document_role` column and a small election pass:

- **Election is candidate-only**, like the LLM step: it enumerates compatible resolvements and picks one or abstains. It never introduces a value Stage 4 did not extract. A value whose key resolved to `document_ref` for the role (`FUND:` -> `document_ref:fund`) is the strongest signal.
- **No precedent tier for roles** — which occurrence is this document's investor is inherently per-document. The underlying entity/period mapping still precedents normally.
- **Re-pointing a role is one write** — moving `document_role` between resolvement rows, guarded by the partial unique index.

**Storage:** `label_resolvements`, `resolvement_candidates`, `precedents`

```text
label_resolvements                      -- one row per label per run
  id                  uuid PK
  run_id              FK extraction_runs
  label_id            FK labels UNIQUE
  status              enum: auto | proposed | confirmed | rejected
  target_kind         enum: canonical_metric | entity | document_ref | period | measure | literal
  target_ref          text (nullable)   -- (scope-qualified) concept / entity id / role name / period strategy / representation
  mint_request        jsonb (nullable)  -- new entity proposal
  method              enum: precedent | llm | human
  precedent_id        FK precedents (nullable)
  document_role       text (nullable)   -- e.g. investor | fund | period_end; validated against
                                        -- registry slot names, not a DB enum. Partial UNIQUE
                                        -- (run_id, document_role) WHERE document_role IS NOT NULL
  confidence          numeric
  rationale           text (nullable)
  resolver_version    text
  created_at          timestamptz

resolvement_candidates                  -- the enumerated options the decision was made among
  id                  uuid PK
  resolvement_id      FK label_resolvements
  rank                int
  target_kind         enum (as above)
  target_ref          text
  score               numeric
  source              enum: precedent_knn | registry_similarity | entity_blocking | recognizer

precedents                              -- THE universal precedent store
  id                  uuid PK
  organization_id     FK (nullable)     -- NULL only for scope = global
  match_key           text              -- the label normalized_text; the only match dimension
  scope               enum: doc_type | org | global
  scope_ref           text (nullable)
  target_kind         enum (as above)
  target_ref          text
  created_from        FK label_resolvements (nullable)
  created_by          FK member (nullable)
  match_count         int
  last_used_at        timestamptz
  revoked_at          timestamptz (nullable)  -- precedents are never deleted, only revoked
  -- one active target per (organization_id, match_key, scope, scope_ref):
  -- partial UNIQUE (...) WHERE revoked_at IS NULL
```

Rules: exact precedent hits auto-accept (`status = auto`); everything from the LLM is `proposed` until a human confirms. One store for metrics and entities (an entity alias is a precedent with `target_kind = entity`). Confirmation mints org-wide by default. `global` is the cross-org seed tier. Precedents are append-only: corrections revoke and re-mint. Roles never gate mapping — a missing role surfaces downstream as a `blocked` proposal, never as silence here.

Emits `run.resolved`, triggers Stage 6.

### 6 Proposals

A proposal is computed, never authored: `proposal = f(carrier payload, resolvements, registry, document roles)`. Stage 6 runs `f()` over every carrier and persists the result plus its input lineage.

This is the pipeline's center of gravity. Stages 3 to 5 stay simple precisely because composition absorbs the difficulty. `f()` is an ordered chain of small pure rules — each unit-testable in isolation with a minimal fixture. Build the per-rule fixture matrix (0/1/2 governing canonicals, scope conflict, mixed-currency, missing unit, series-identity pass/fail) up front.

A cell's governing labels are its full row-axis ancestor chain + full column-axis ancestor chain + the table's title label (+ document roles for slots nothing local fills). Each resolved target contributes one component of the fact:

- **Base concept** — exactly one governing label must resolve to a `canonical_metric`, wherever it is (leaf header, spanning parent, other axis, or table title). Zero -> residue; two -> `blocked` with a flag. Prose-level context (currency, period, "in thousands") resolves to a label and joins the chain as a default; per-column meaning encoded solely in prose is left unbound (loud residue, review binds candidate-only).
- **Temporal scope** — from the qualified canonical if the label embedded it, else a governing `period` resolvement, else the registry default. Label-embedded scope wins over axis-derived; disagreement flags `scope_conflict`.
- **As-of date** — cumulative/instant metrics are read as of a point in time. The document's transaction resolves to a date; "after this notice" places cells at that as-of, "prior to this notice" at the as-of immediately before. Two cumulative figures compose into the same metric at two as-of dates.
- **Representation** — from a governing `measure` resolvement, cross-checked against the atom's unit and the registry. Absent -> default `value`.
- **Unit + scale** — fall-through: atom's own unit -> nearest governing label's `unit_hint` -> document-level currency -> flag `missing_unit`. Mixed currency defeats the document default deliberately: when siblings carry differing `unit_hint`s, an un-hinted atom flags `missing_unit` rather than inheriting a default. FX is never computed.
- **Entity slots + when** — axis labels, then document roles. A governing `document_ref` label fills its slot by dereferencing the document role; the investor/fund table split falls out of ordinary slot-filling.
- **Cross-fact validation** — the prior/current/after triple is a time-series identity: `cumulative(before) + flow(this) = cumulative(after)`. Persist the "after" stock and the current-drawdown flow (both carry absolute dates); the "prior" figure is a corroborating input, not an independent row. A broken identity flags `series_identity_mismatch`.

**Fact identity = base concept + temporal scope + slots + when (+ representation).** This lets "Capital Called" under both a QTD and an ITD column compose into two cleanly distinct facts. The proposal body is the only JSONB in Postgres — a disposable projection rebuildable from `proposal_inputs`.

**Storage:** `proposals`, `proposal_inputs`

```text
proposals                               -- recomputable projection of f(); one row per computed proposal
  id                  uuid PK
  run_id              FK extraction_runs
  identity            text              -- stable hash over payload lineage; survives recompute, keys the decision log
  kind                enum: entity | metric
  body                jsonb             -- computed proposal; the only JSONB in Postgres, disposable
  body_hash           text
  status              enum: ready | blocked | flagged   -- blocked: missing required slot (loud)
  flags               text[]            -- missing_required_slot:investor, scope_conflict, ...
  compute_version     text
  computed_at         timestamptz
  UNIQUE (run_id, identity)

proposal_inputs                         -- the explicit computation audit: exactly what f() consumed
  proposal_id         FK proposals
  input_kind          enum: cell | freestanding_value | resolvement | registry_version
  input_ref           text
  role                text              -- value | metric_label | entity_slot:investor | period | ...
```

Rules: recompute, not sync — editing a resolvement re-runs `f()` for affected proposals in place. Loud failure — a position metric missing its investor slot is `blocked`, never silently reshaped. Dedup/corroboration happens here. Every atom/value label must end up referenced by a proposal or be explicitly marked dropped (the coverage invariant); residue is a run-level report, not silence. Relationship edges (`invests_in`) are not proposals in v2 — a position metric names both endpoints, so the edge is upserted as a side effect at commit.

Emits `run.proposed`. Review is ready.

### 7 Commit

Approval turns a proposal into truth. Decisions are append-only and keyed on the proposal's stable `identity`, not its row id, so they survive recompute and re-extraction.

- Approve commits the truth rows and the decision row in one transaction. Each truth row carries `decision_id`, so truth -> decision -> proposal -> inputs -> cell -> parse -> source is one unbroken provenance walk.
- Staleness is derived. If a decided proposal's recomputed `body_hash` no longer matches `body_hash_at_decision`, it is flagged stale and re-queued.
- Reverts append a decision and reverse the truth write. Nothing is deleted.

**Storage:** `proposal_decisions`

```text
proposal_decisions                      -- append-only; the durable human/agent record
  id                  uuid PK
  organization_id     FK
  proposal_identity   text              -- joins to proposals.identity
  kind                enum: approve | reject | edit
  actor               FK member (or agent identifier)
  body_hash_at_decision text            -- what the human actually saw
  edited_body         jsonb (nullable)  -- present on edit
  created_at          timestamptz
```

Emits `proposal.committed`.

## The review screen

One screen per source, one queue, reading the relational substrate directly. The queue mixes item types (filterable), ordered so upstream decisions come first — confirming a label mapping collapses the cell-level items beneath it:

1. **Label mappings** — labels with `proposed`/missing resolvements, grouped by normalized form, prefilled with the LLM pick + candidates + rationale. Confirming writes the resolvement, mints the precedent, and recomputes affected proposals live.
2. **Document roles** — the resolvements carrying a `document_role`, shown with their pointed-at value and page/bbox highlight; re-pointing recomputes downstream. A filtered view of the resolvement queue, no separate anchor screen.
3. **Freestanding values** — non-table values with their optional key labels and payloads; mostly confirmation reading.
4. **Bulk approve tables** — the grid in document geometry, axis labels as chips, cells showing computed proposals; approve-all commits every `ready` proposal in the table.

## Bundle splitting

A source is one logical document. A PDF containing several statements is not split. This is load-bearing: the partial unique index `(run_id, document_role)` allows exactly one investor, fund, and period_end per run, so a bundle of several investor statements is structurally unrepresentable. The clean seam to retrofit splitting is between Stage 1 and Stage 2: a `documents` layer so one source fans out into N logical documents, each parsed independently. A single-document source is the degenerate N=1 case.
