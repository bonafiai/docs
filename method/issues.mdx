---

An issue is where spec-driven work starts. It lives in Linear, not the repo (the repo's first feature file is spec-kit's `spec.md`). You describe a feature, an agent files it, you review, no hand-written tickets.

<Steps>
  <Step title="Describe">
    Tell an agent the feature in plain language, the behavior you want and why. No format to follow, that is the agent's job.
  </Step>
  <Step title="File">
    The agent opens the issue in Linear through the [Linear MCP](/stack/tooling), building intent, context, and scope from your words plus the chat and codebase. The repo [contract](/stack/tooling) requires this, so every agent files consistently.
  </Step>
  <Step title="Review">
    Check the filed issue, do not write it. Done when intent reads as behavior not implementation, every source a cold reader needs is linked, and scope is bounded with an explicit "not now".
  </Step>
</Steps>

A filed issue is the loop's only input, an agent picks it up and runs spec-kit from there, see [Workflow](/method/workflow).
