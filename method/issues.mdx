---
title: "Issues"
description: "The unit of work, filed and kept current by agents."
---

Agents file and work issues from chat and codebase context, humans hand over intent. We move fast by creating issues through agents, not by writing tickets by hand. The bar every issue clears: any agent can pick it up cold.

| Name | Description |
| --- | --- |
| Intent | What and why in behavior terms. For features this becomes the [`/speckit.specify`](https://github.com/github/spec-kit) input |
| Context | The links a cold agent needs: docs pages, official sources, related issues and PRs, the originating chat |
| Scope | What this issue is not, the guard against over-eagerness |

## Kinds

Not every issue is a feature. The kind decides the path:

| Name | Description |
| --- | --- |
| Feature | New behavior, takes the full loop, see [Workflow](/method/workflow) |
| Fix | Regression or tweak, light path, names the broken scenario |
| Chore | Dependencies, tooling, docs, no product behavior |

## Lifecycle

| Name | Description |
| --- | --- |
| Filed | An agent opens it the moment work is identified, carrying intent, context, and scope |
| Worked | The agent posts decisions and progress on the issue as they happen, so it never goes stale |
| Closed | The PR links the issue, leftovers become new issues, scope is never silently dropped |

Done is not the agent saying done, done is the gate, see [Testing](/method/testing).
