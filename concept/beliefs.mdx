These beliefs explain why the rules and workflow exist. Everything else in this docs site implements them.

## 1. Platform engineering for agents

Your job is shifting from building the feature to building the system that lets an agent build the feature.

We invest in internal tooling relative to team size: versioned MCP servers, pinned skills, spec commands, CI gates, and deploy loops the agent can run. Even as a small team, it feels like having hundreds of engineers, because the harness does the repetitive loop.

## 2. Spec-driven development with verified behavior

Specifications define how the code should behave. Tests verify the agent actually implemented that behavior, so you are not back in the codebase checking by hand.

Every feature spec carries Gherkin-style acceptance scenarios. Each scenario maps to one end-to-end test. Done means scenarios pass, not "looks right".

## 3. Minimalism as discipline

Few object classes. One machine per concern. No parallel near-duplicate concepts.

When two things look similar, unify them. When something can be derived, do not store it. When a stage can be replayed, never mutate, append a new run.
