Testing is how we know the agent built what the spec said. Every feature spec writes its acceptance scenarios against this contract.

## Five layers

| Layer | What it catches |
| --- | --- |
| Static | Syntax, types, formatting (oxlint, prettier, `tsc`) |
| Invariant scripts | Repo rules made executable, run inside `bun run lint` |
| Unit | Pure logic and components (Vitest, no network, no model, no env) |
| Integration | Database, migrations, tenant boundaries (Vitest against real Postgres in CI) |
| E2E acceptance | User-visible behavior from specs (Playwright) |

Invariant scripts include the truth-write guard, the spec-reference check, the scenario-coverage check, and the rules-mirror hash check.

E2E acceptance is the layer that makes spec-driven development verifiable: one spec scenario = one Playwright test, scenario id in the test title, run against a production build with seeded fixtures.

## The PR gate, six checks

lint, typecheck, unit, integration, e2e-acceptance, spec-reference. All deterministic, all key-free.

Out of the gate: LLM evals (nondeterministic, paid, nightly or on-demand only), coverage percentages, visual snapshots. Docs-only PRs pass through a companion workflow so they never wedge.

<Warning>
The gate does not grow by default. A seventh check needs a written reason.
</Warning>

## Libraries

| Choice | Why |
| --- | --- |
| Vitest | Bun-compatible, proven in the codebase, jsdom and Testing Library for components |
| Playwright | The E2E standard, trace and debug tooling, Gherkin scenario ids in test titles |
